一、数据抽象

  1. 什么是数据抽象
    数据抽象是对实际的人、物、事和概念进行人为处理，抽取所关心的共同特性，忽略非本质的细节，并把这些特性用各种概念精确地加以描述，这些概念组成了某种模型。

  2. 什么是数据类型的接口
    接口泛指实体把自己提供给外界的一种抽象化物（可以为另一实体），用以由内部操作分离出外部沟通方法，使其能被内部修改而不影响外界其他实体与其交互的方式。

  3. Java语言如何限定类型的接口？

  • Java接口是一系列方法的声明，是一些方法特征的集合，一个接口只有方法的特征没有方法的实现，因此这些方法可以在不同的地方被不同的类实现，而这些实现可以具有不同的行为（功能）。
  • 在Java中，你可以使用泛型（Generics）来限定接口的类型。通过泛型，你可以在接口中定义参数化类型，使得接口可以与不同类型的数据一起使用，提高代码的灵活性和类型安全性。
  1. 数据抽象与面向对象之间的关系？

数据抽象：

  • 数据抽象是一种编程思想，它强调将数据的表示和数据的操作分离，将关注点放在数据的本质和行为上。
  • 抽象是通过隐藏不必要的细节、提取关键特征，将复杂的现实问题简化为程序设计中的模型。这种抽象使得程序更易于理解、维护和扩展。
  • 在面向对象的编程中，数据抽象通过类和对象来实现，类定义了对象的属性（数据）和方法（行为），并通过访问控制来限制对数据的直接访问。

面向对象：

  • 面向对象是一种编程范式，它以对象为中心，将数据和相关操作封装在一起，通过定义类和对象来组织和管理代码。
  • 面向对象编程的核心思想包括封装、继承和多态。封装使得对象的内部实现对外部隐藏，继承允许新的类继承和重用已有类的特性，多态允许不同类的对象对同一消息做出不同的响应。
  • 面向对象的方法使得程序更加模块化，提高了代码的重用性和可维护性。

关系：

  • 面向对象编程通过类和对象的抽象概念，实现了数据的抽象。类定义了数据的结构和行为，对象则是类的实例，包含了具体的数据值。
  • 数据抽象通过隐藏底层实现细节，暴露出对外接口，使得使用者可以关注对象的高层行为而不必关心内部实现。这种抽象有助于降低复杂性，提高代码的可理解性和可维护性。

二、LSP

面向对象的三大原则？

  1. 单一职责原则（Single Responsibility Principle，SRP）：
    • 一个类应该只有一个引起变化的原因，即一个类应该只有一个责任。
    • 这意味着一个类应该只有一个引起它变化的原因，这样如果需要修改类，只有一个地方需要修改。这有助于降低耦合性，提高类的可维护性。
  2. 开闭原则（Open/Closed Principle，OCP）：
    • 软件实体（类、模块、函数等）应该对扩展开放，对修改关闭。
    • 这意味着当需要添加新功能时，不应该修改已有的代码，而是通过扩展现有代码来实现。这样可以保持现有代码的稳定性，降低引入新功能时的风险。
  3. 里氏替换原则（Liskov Substitution Principle，LSP）：
    • 所有引用基类的地方必须能够透明地使用其子类的对象，即子类必须能够替代基类。
    • 这意味着继承关系应该是一个 “is-a” 关系，子类应该能够完全替代父类而不引起问题。遵循 LSP 有助于确保代码的稳定性和可靠性。

什么是里氏替换原则（LSP）及其意义?

  1. 要更直观理解 LSP，可以简化为以下两个要点：
  • 子类必须保持父类的行为： 子类应该能够替代父类而不改变程序的正确性。即，对于任何父类被用到的地方，都可以用其子类来替代而不引起错误。
  • 保持一致的接口： 子类必须保持与其父类一致的接口。这意味着子类应该实现父类定义的所有方法，同时保持方法签名和预期行为。
  1. LSP 的意义主要体现在以下几个方面：
  • 可替代性： LSP确保派生类可以替代其基类，而不引起使用基类的代码出现错误。这意味着代码可以更加灵活，可以使用基类的地方都能够使用派生类，从而实现多态性。
  • 稳定性和一致性： LSP强调子类型必须保持与其父类型一致的接口和行为。这样一来，使用基类的代码不需要关心具体是哪个子类，只需要知道它们可以替代基类，这保持了程序的稳定性和一致性。
  • 代码的可维护性： 遵循LSP有助于提高代码的可维护性。当系统需要扩展时，可以通过添加新的子类而不影响现有代码，从而降低了系统的耦合度，使得系统更易于理解、扩展和维护。

从LSP解释什么是继承，什么是多态？

  1. 继承：
    • 继承是面向对象编程中的一种机制，允许一个类（子类或派生类）继承另一个类（父类或基类）的属性和方法。子类可以重用父类的代码，同时可以添加、修改或覆盖父类的行为。
    • LSP强调了继承关系中的可替代性，即子类应该能够替代其基类而不引起程序错误。这意味着在使用继承时，子类应该保持与其父类一致的接口和行为，以确保对基类的使用在替代为子类时不会产生意外结果。
  2. 多态：
    • 多态是面向对象编程中的一个重要概念，指的是同一操作对于不同对象可以有不同的解释。多态允许同一个方法名在不同的类中具有不同的实现，使得代码更加灵活、可扩展和可维护。
    • 在继承关系中，多态表现为通过基类的引用或指针调用派生类的方法。基类定义了一个共同的接口，而不同的子类实现了该接口的不同版本。使用基类的引用或指针调用方法时，实际执行的是对应子类的方法。
    • LSP对多态的实现有着直接的影响，因为只有当子类能够替代其基类时，多态才能够正常工作。如果子类不符合基类的契约，可能导致多态引发错误。

继承和多态是面向对象编程中两个重要的概念，它们通过LSP关注了继承关系中的正确性和稳定性。通过遵循LSP，继承关系中的子类可以更自由地替代其基类，从而实现了多态性。这样的设计使得代码更具灵活性，能够适应变化，并保持了良好的可维护性。

介绍遵循LSP的四种形式？

遵循里氏替换原则（Liskov Substitution Principle，LSP）的四种形式是由Barbara Liskov在其原始论文中提出的。这四种形式帮助解释了LSP的具体要求，确保子类型可以有效地替代其基类型。这些形式包括：

  1. 协变返回类型（Covariant Return Types）：
    • 如果S是T的子类型，那么S的方法的返回类型必须是T方法的返回类型的子类型。
    • 换句话说，子类的方法可以返回父类方法的返回类型的子类型，但不能返回父类方法返回类型的超类型。
  2. 逆变参数类型（Contravariant Parameter Types）：
    • 如果S是T的子类型，那么S的方法的参数类型必须是T方法的参数类型的超类型。
    • 这意味着子类的方法可以接受比父类方法更广泛的参数类型，但不能接受比父类方法更狭窄的参数类型。
  3. 协变参数类型和逆变返回类型（Covariant Parameter Types and Contravariant Return Types）：
    • 这是协变参数类型和逆变返回类型两者的结合。如果S是T的子类型，那么S的方法的参数类型必须是T方法的参数类型的超类型，并且S的方法的返回类型必须是T方法的返回类型的子类型。
  4. 不变类型（Invariant Types）：
    • 如果S是T的子类型，那么S的方法的参数类型和返回类型必须与T方法的参数类型和返回类型相同，不能有任何类型变化。
    • 这是最保守的形式，要求子类的方法参数和返回类型与父类的完全一致。

三、抽象依赖原则

为什么要有这个原则以及其定义？

抽象依赖原则（Dependency Inversion Principle，DIP）是SOLID原则中的一项，它强调高层模块不应该依赖于低层模块，两者都应该依赖于抽象。

  1. 定义:
  • 高层模块不应该依赖于低层模块。两者都应该依赖于抽象。
  • 抽象不应该依赖于细节。细节应该依赖于抽象。
  1. 这个原则的提出是为了解决软件设计中的一些常见问题：
  • 灵活性和可维护性： 当高层模块直接依赖于低层模块时，系统的灵活性和可维护性会降低，因为对低层模块的修改可能会影响到高层模块。
  • 复用性： 直接依赖于具体实现会导致代码的耦合性增加，从而降低了模块的可复用性。
  • 测试性： 当高层模块直接依赖于低层模块时，测试高层模块可能会变得更加困难，因为必须考虑到低层模块的具体实现。

通过使用抽象依赖原则，我们可以将高层模块与低层模块解耦，并通过引入抽象来减少模块之间的直接依赖。这样可以提高系统的灵活性、可维护性和可测试性，同时提高代码的复用性。

在实践中，抽象依赖原则通常与依赖注入（Dependency Injection）一起使用，通过将依赖关系的创建和维护委托给外部容器，实现了依赖关系的反转。这使得系统更容易扩展和维护。

抽象依赖原则的使能工具有什么特点？

抽象依赖原则的实现通常涉及使用依赖注入（Dependency Injection，DI）或依赖反转容器（Dependency Inversion Container）等使能工具。这些工具有一些特点，有助于在软件设计中实现抽象依赖原则：

  1. 依赖注入： 依赖注入是实现抽象依赖原则的一种方式，它通过将依赖关系从高层模块传递给低层模块，从而避免了高层模块直接依赖于具体的低层模块。依赖注入可以通过构造函数注入、方法注入、属性注入等方式来实现。
  2. 接口和抽象类： 使用接口和抽象类定义抽象，通过依赖注入将具体实现注入到高层模块。这样，高层模块只依赖于抽象而不依赖于具体实现。
  3. 依赖反转容器： 依赖反转容器是一种实现抽象依赖原则的工具，它负责创建和维护对象之间的依赖关系。容器将控制权从高层模块转移到容器，从而实现了依赖的反转。
  4. 配置： 使用配置文件或其他外部配置方式，将具体实现类的信息从代码中分离出来。这使得在不修改代码的情况下可以更改具体实现，从而实现了抽象依赖原则。
  5. 接口隔离： 遵循接口隔离原则，定义细粒度的接口，使得高层模块只依赖于其需要的接口，而不依赖于不必要的接口。这有助于减小模块之间的耦合。
  6. 依赖查找： 通过依赖查找的方式，高层模块可以向容器请求需要的依赖。这种方式可以在运行时动态地解析和获取依赖关系。

这些工具的共同特点是通过降低模块之间的直接依赖关系，使得高层模块不依赖于具体的低层模块，而是依赖于抽象。这样可以提高系统的灵活性、可维护性和可测试性。选择合适的工具取决于具体的项目需求和架构设计。

使能工具被称为控制反转工具容器，为什么控制反转工具容器与控制反转没有关系？

  1. “控制反转”（Inversion of Control，IoC）和”依赖注入”（Dependency Injection，DI）是两个相关但不同的概念。
  • 控制反转通常指的是将控制权从一个模块或类转移到另一个模块或类，
  • 而依赖注入是实现控制反转的一种方式。
  • 使能工具通常被称为”控制反转容器”或”依赖注入容器”，它们并不直接涉及到控制反转，而是实现了依赖注入，即将依赖关系从高层模块传递给低层模块。
  1. 为什么控制反转容器与控制反转没有直接关系呢？
  • 这是因为”控制反转”一词的广义概念包括了依赖注入，但并不局限于依赖注入。
  • 控制反转强调的是控制权的转移，而依赖注入是一种实现控制反转的具体方式。

  1. 控制反转容器的主要作用是管理和维护对象之间的依赖关系，以及在需要时为对象提供依赖。它通过读取配置、扫描注解等方式，实现了对象的创建、装配和注入，从而实现了依赖注入。在这个过程中，容器负责控制对象的创建和依赖的管理，而不是由对象自己控制。

  2. 综上所述，尽管控制反转容器实现了依赖注入，但控制反转并不等同于依赖注入，而是一个更广泛的概念。因此，控制反转容器被称为”控制反转容器”是因为它实现了控制反转的一部分，即将依赖关系的控制权从对象自身转移到了容器

    五、函数式编程范式

    函数式编程范式核心观念

函数编程范式的核心观念：以λ表达式构造一切。

  1. 什么是λ表达式呢？普通的数学函数如f(x)=x+1，功能是给其参数x加上1。为了将数学函数表示成计算机界常用的表达式，可以换一种写法：λx.(x+1)，读成“对于参数x，x+1”(假定操作符+，数字1已经被定义)。

  2. 合法的λ表达式的递归定义如下：

  • 变量(和文字) 是一个λ表达式。
  • 操作符和λ表达式组成的表达式，是λ表达式。(可以归入函数应用)
  • 函数定义和函数应用是λ表达式。
  1. 特性：
  • 数据的不变性。
  • 使用递归作为循环的机制。
  • 函数没有副作用和引用透明。
  • 支持闭包(closures)和高阶函数(higher-order functions)。
  • 支持懒惰计算(lazy evaluation)。

什么是lambda表达式？

  1. 定义：
    它是一个匿名函数，Lambda表达式基于数学中的λ演算得名，直接对应于其中的lambda抽象(lambda abstraction)，是一个匿名函数，即没有函数名的函数。Lambda 是一个匿名函数，可以把 Lambda表达式 理解为是一段可以传递的代码 (将代码像数据一样进行传递)。可以写出更简洁、更灵活的代码。

  2. 在丘奇的λ演算中，函数定义有两个特点。

  • λ表达式都是匿名函数。函数定义指构造一个函数，W是参数为变量x的λ表达式，则λx . W是λ表达式，此时W是函数体，参数就是变量x。例如(lambda (x) (* x x))描述了一个计算过程。为匿名函数取一个名字，或者说定义一个变量，能够方便地多次使用该函数，通常会受到编程语言的支持。
  • 每个函数只有一个输入参数，如λn. λm.λf.λx**.** ((n f) (++ m) )。而在编程实践中，函数通常可以有一个参数列表。逻辑学家 Haskell Curry证明可以将一个拥有多个参数的函数转化为只有一个参数的多个函数的连续调用，这一转化过程称为对拥有多个参数的函数的currying/柯里化。

为什么Scame语言里用递归实现循环？

有几个原因解释了为什么在 Scheme 中更倾向于使用递归来实现循环：

  1. 一致性： 在 Lisp 系列语言中，递归与其他形式的迭代一样被视为一种迭代控制结构，这带来了一致性。在 Scheme 中，递归和迭代都被视为迭代过程，通过 do 循环、let 形式和递归等方式，可以以一致的方式实现各种迭代。
  2. 简洁性和表达力： 递归的表达力更强，有时可以更清晰地描述算法。通过递归，程序员可以更直接地表达问题的本质，而无需引入很多额外的控制结构。
  3. 尾递归优化： Scheme 支持尾递归优化，这是一种递归的特殊形式，可以通过编译器优化为迭代。尾递归优化避免了递归时栈溢出的问题，使得递归在 Scheme 中更加高效。
  4. 函数式编程风格： Scheme 是一门函数式编程语言，它鼓励使用纯函数、不可变数据以及递归。递归更符合函数式编程的风格，而循环则更强调可变状态。
  5. 灵活性： 递归允许在不引入可变状态的情况下进行循环，这符合 Scheme 的函数式编程理念。在某些情况下，递归可以提供更灵活和易于理解的解决方案。

列举函数式编程语言中的常见的特点？

  1. 纯函数（Pure Functions）：
    • 函数的输出只由输入参数决定，不依赖于外部状态或可变数据。纯函数没有副作用，执行相同的输入总是产生相同的输出。
  2. 不可变性（Immutability）：
    • 数据一旦被创建，就不能被修改。不可变性有助于避免副作用、简化并发编程，以及提高程序的可靠性和可维护性。
  3. 高阶函数（Higher-Order Functions）：
    • 函数可以接受其他函数作为参数，也可以返回函数作为结果。高阶函数支持函数作为一等公民，可以传递、赋值和使用它们。
  4. 透明性（Referential Transparency）：

什么是框架？介绍框架和工具箱之间的不同

框架（Framework）：
一个框架是一个预先编写好的可重用的代码，它提供了一种通用的结构，以便在这个结构中插入自定义代码，从而解决特定领域的问题。框架通常包含了一系列设计模式、标准实践和工具，以便帮助开发人员更容易地构建和维护软件。框架通常定义了应用程序的骨架，开发者需要根据框架的规范填充具体的业务逻辑。框架的目标是提供一个可扩展、可维护、可重用的基础架构，以促进软件开发的效率和一致性。

工具箱（Toolkit）：
工具箱是一组工具的集合，这些工具通常是独立的、可重用的软件组件，用于执行特定的任务。工具箱提供了一组工具，开发人员可以根据需要选择和组合这些工具，以满足特定的开发要求。工具箱通常不像框架那样具有完整的应用程序结构，而是为了提供一系列单独的功能而设计。

框架和工具箱之间的不同：

  1. 结构 vs. 工具：
    • 框架是一个结构，提供了一个完整的应用程序的基础架构和设计。
    • 工具箱是一组独立的工具，每个工具解决特定的任务或提供特定的功能。
  2. 约束性 vs. 自由度：
    • 框架通常具有一定的约束性，要求开发者按照框架的规范来构建应用程序。
    • 工具箱提供更大的自由度，开发者可以根据需要选择和组合工具。
  3. 提供性 vs.使用性：
    • 框架提供了应用程序的骨架，开发者需要填充具体的业务逻辑。
    • 工具箱提供了独立的工具，开发者可以选择性地使用这些工具。
  4. 全面性 vs. 单一性：
    • 框架通常是全面的，提供了一个完整的解决方案，包括应用程序的结构和设计。
    • 工具箱通常是单一性的，每个工具解决一个具体的问题。
  5. 约束性编程 vs.自由编程：
    • 框架可能要求开发者按照框架的规范进行约束性编程。
    • 工具箱通常更灵活，允许开发者采用更自由的编程方式。

介绍一下控制反转、怎样认识和理解控制反转

  1. 控制反转（Inversion of Control，IoC）是一种软件设计思想，它将传统的程序设计流程颠倒过来，将控制权从程序自身转移到外部容器或框架，以实现更灵活、松耦合的代码结构。
  2. 在传统的程序设计中，应用程序通常控制和调用框架、库或组件，即应用程序是主动的，而框架或库是被动的。而在控制反转中，控制权被颠倒了，框架或容器成为了主动的元素，负责管理和控制应用程序的执行流程。

介绍java里回调函数的概念及其特点

在Java中，回调函数是一种常见的编程模式，它允许一个对象（”回调函数”）在另一个对象（”调用者”）发生特定事件或条件满足时被调用。回调函数是一种实现事件驱动编程的方式，它允许程序在异步事件发生时执行某些操作。

  1. 回调函数的概念：
  • 回调函数的基本思想是将一个可执行的代码块传递给某个函数（或对象），以便在未来某个特定的事件发生时执行。这种方式使得程序可以更加灵活地处理异步操作或特定的事件。
  • 在Java中，回调函数通常通过接口实现，将一个实现了特定接口的对象传递给调用者，调用者在某个事件发生时调用该接口的方法。
  1. 回调函数的特点：
  • 松耦合性（Loose Coupling）： 回调函数使得调用者和回调函数的实现彼此之间更加解耦。调用者不需要知道回调函数的具体实现，只需要知道回调函数遵循了某个接口。
  • 事件驱动（Event-Driven）： 回调函数通常用于处理事件驱动的场景。例如，当某个按钮被点击、网络请求完成、定时器触发等事件发生时，注册的回调函数将被调用。
  • 异步编程： 回调函数是处理异步编程的一种方式。通过注册回调函数，程序可以继续执行其他任务，而不必等待事件完成。
  • 扩展性（Extensibility）： 回调函数使得程序更容易扩展。可以轻松地替换或添加新的回调函数，而不必修改调用者的代码。

回调机制与框架关系

回调机制是框架中用于实现事件驱动、异步编程和松耦合性的一种关键机制。框架通过提供回调接口和事件触发机制，为开发者提供了一种灵活且可扩展的方式来定制和扩展框架的行为。
以下是回调机制与框架关系的一些重要方面：

  1. 事件驱动框架： 许多框架是事件驱动的，这意味着框架中的某些功能或行为是由事件触发的。回调机制通过让开发者注册回调函数来响应这些事件，从而实现了事件驱动。
  2. 框架提供回调接口： 框架通常会定义一些回调接口，开发者可以实现这些接口，并通过注册的方式将自己的回调函数传递给框架。框架在特定事件发生时调用这些注册的回调函数。
  3. 松耦合性： 回调机制在框架中促使了松耦合性的设计。框架不需要知道回调函数的具体实现，只需知道回调函数遵循了特定的接口。这使得框架更加灵活和可扩展。
  4. 异步编程： 在异步编程中，回调机制是一种常见的方式。例如，当异步任务完成时，框架可以调用注册的回调函数来处理结果，而不是等待任务完成。
  5. 观察者模式： 回调机制在观察者模式中得到了广泛的应用，观察者模式是一种定义了对象之间一对多依赖关系的设计模式。观察者（回调函数）注册到主题（框架）上，当主题发生变化时，通知所有注册的观察者。
  6. 事件处理框架： 一些框架专门用于事件处理，它们通过回调机制让开发者注册事件处理函数。这样，当特定事件发生时，框架会调用相应的回调函数。
  7. Web框架和GUI框架： 在Web开发和图形用户界面（GUI）开发中，回调机制也是常见的。例如，在Web框架中，可以注册处理HTTP请求的回调函数；在GUI框架中，可以注册处理按钮点击事件的回调函数。

九、好莱坞法则

什么是好莱坞法则 p34

好莱坞法则的关键点包括：

  1. 高层模块决定执行流程： 在系统设计中，高层模块（或上层组件）决定了整体的执行流程和控制权。低层模块只负责实现特定的功能，由高层模块来调用和控制。
  2. 低层模块提供服务： 低层模块负责提供特定的服务或实现具体的功能，但它们不决定何时、如何被调用。调用的时机和方式由高层模块来决定。
  3. 减少依赖关系： 通过将控制流程的权力集中在高层模块，可以减少系统中各个模块之间的直接依赖关系。这种减少依赖的设计有助于提高系统的灵活性和可维护性。
  4. 反馈机制： 低层模块向高层模块提供反馈，以便高层模块能够了解低层模块的执行状态或结果。这种反馈机制有助于高层模块更好地控制和协调系统的行为。

说明回调机制与好莱坞法则的关系

回调机制和好莱坞法则共同强调了高层组件对于低层组件的控制权，以实现更加灵活、可扩展和可维护的设计。在实际的软件开发中，它们经常结合使用，特别是在事件驱动和异步编程的场景中。

  1. 控制反转（Inversion of Control）： 回调机制是一种实现控制反转的方式。在回调机制中，高层组件将一个低层组件（回调函数）传递给某个接口或框架，使得低层组件在特定事件发生时被调用。这是一种反向的控制流程，即高层组件决定何时调用低层组件。
  2. 好莱坞法则的体现： 在回调机制中，好莱坞法则的体现在于高层组件对低层组件的控制。高层组件决定了执行流程，它负责调用低层组件的回调函数，而低层组件则负责提供特定的服务或实现特定的功能。
  3. 减少依赖关系： 好莱坞法则通过减少高层组件对低层组件的直接依赖关系，使系统更加灵活。回调机制通过将低层组件的执行控制权交给高层组件，也有助于减少依赖关系，实现了松耦合的设计。
  4. 反馈机制： 在回调机制中，低层组件向高层组件提供反馈，通常是通过回调函数的参数或回调接口的方法。这种反馈机制使得高层组件能够了解低层组件的执行状态或结果，符合好莱坞法则中的“我们会打电话给你”的思想。
  5. 框架中的应用： 在一些框架和库中，回调机制和好莱坞法则常常结合使用，以实现可扩展、可定制的设计。例如，事件驱动框架中，注册事件处理函数就是一种回调机制，并且这些函数通常是在框架内被调用，体现了好莱坞法则。

模板方法模式适用于以下两种主要场景：

  1. 算法框架定义： 当一个算法的执行流程在各个子类中基本相同，但其中某些步骤的具体实现可能不同，可以使用模板方法模式。在这种情况下，父类定义一个算法的框架，将具体的实现延迟到子类中。子类通过实现或覆盖父类的抽象方法来提供算法中特定步骤的实现。
  2. 代码复用： 当有多个子类中存在相似的行为，而且这些行为可以被抽象到一个共同的父类中，以实现代码复用和避免重复实现时，可以使用模板方法模式。通过将共同的行为放在父类中，避免了在每个子类中重复编写相同的代码。

模板方法模式适用于具有共同执行流程但某些步骤可能有不同实现的场景，以及需要通过代码复用来避免重复实现的场景。

模板模式意图

  1. 定义算法的框架： 模板方法在父类中定义了算法的框架，确定了算法执行的步骤和顺序。这些步骤可以包含抽象方法、具体方法和钩子方法。
  2. 延迟实现到子类： 具体步骤的实现被延迟到子类中。父类中的抽象方法由子类提供具体的实现，从而使得不同的子类可以实现相同的算法的不同步骤。
  3. 保护算法结构： 模板方法通过将算法的关键步骤放在模板方法中，保护了整个算法的结构。子类只需要关注实现自己特定的步骤，而不需要担心算法的整体结构。
  4. 提高代码复用性： 模板方法模式有助于提高代码的复用性。算法的框架由父类提供，而具体步骤的实现由子类完成，这使得相同的算法框架可以在不同的子类中重复使用。
  5. 反向的控制结构： 模板方法导致了一种反向的控制结构，即由高层模块（父类）来控制整个算法的执行流程，而低层模块（子类）负责实现具体的步骤。
  6. 扩展性： 模板方法模式提供了一种扩展的机制。通过添加新的子类并实现特定步骤的具体实现，可以轻松扩展算法，而不需要修改模板方法的结构。

探讨模板方法模式中可变部分的设计方案

在模板方法模式中，可变部分指的是算法中具体步骤的实现，这部分由子类来负责。设计这部分时，需要考虑如何在父类中定义框架，同时保留足够的灵活性，使得子类可以根据自己的需要进行定制。

  1. 抽象方法： 在父类中声明抽象方法，由子类负责具体实现。这是模板方法模式的核心特点，子类必须提供对应的实现。抽象方法强制子类提供某些具体步骤的实现，确保了算法的一致性。
  2. 具体方法（可选重写）： 在父类中提供一些具体实现的方法，子类可以选择性地重写这些方法。这样，子类可以选择性地覆盖某些步骤的实现，而不是完全从头开始实现整个算法。
  3. 钩子方法（默认实现）： 在父类中提供一些默认实现的方法，子类可以选择性地覆盖这些方法。这样，子类可以根据需要覆盖或者直接使用父类提供的默认实现。
  4. 回调机制： 使用回调机制，允许子类提供一些回调函数，父类在适当的时机调用这些回调函数。这样，子类可以通过回调来定制算法中的某些步骤。

钩子 相信大家曾经有听到过，但是可能一直不太理解，它是一种被声明在抽象类中的方法，但只有空的或者默认的实现。 钩子的存在，可以让子类有能力对算法的不同点进行挂钩。要不要挂钩由子类自行决定。

增量扩容：
Go采用的是增量扩容方案，当map开始扩容后，每一次map操作都会触发一部分扩容搬迁工作（每进行一次赋值，会做至少一次搬迁工作）。由hmap中的nevacuate成员记录当前的搬迁进度。

扩容情况一：存储的键值对数量过多

  • 这种情况下map会进行翻倍扩容。
  • Go创建一个新的buckets数组，这个buckets数组的容量是旧buckets数组的两倍，并将旧数组的数据逐步迁移至新数组。
  • 旧的buckets数组不会被直接删除，而是会把原来对旧数组的引用去掉，让GC来清除内存。

扩容情况二：溢出桶数量过多

  • 如果出现了这种情况，可能是因为哈希表里有过多的空键值对，很多桶的内部出现了空洞（装不满）。这个时候就需要通过map扩容做内存整理。目的就是为了清除bmap桶中空闲的键值对。
  • 这种情况下map扩容步骤与情况一基本相同，只不过扩容后map容量还是原来的大小。Go会创建一个与原buckets数组容量相同的buckets数组，并将旧数组的数据逐步迁移至这个新数组。再去除旧数组的引用，让GC来清除内存。

GC是什么？为什么要有GC？

GC 是 garbage collection 的缩写,意思是垃圾回收。

  • GC是编程语言中提供的自动的内存管理机制，自动释放不需要的内存对象，让出存储器资源。
  • GC过程中无需程序员手动执行。GC机制在现代很多编程语言都支持(Java/python/golang)，GC能力的性能与优劣也是不同语言之间对比度指标之一。

前面说现在很多语言都有GC机制，但是在我们大一学的c++却是没有的一个标准库，所以在当时我们要自己创建内存空间和销毁，可以使我们更好了解底层原理。

但是全部都要我们程序员自己造轮子，这也太麻烦了。且内存处理是开发人员容易出现问题的地方，所以现在很多语言都提供了自己的垃圾回收装置

基础概念

一个标准进程的内存模型

  • code area（方法区）
  • static area 静态变量
  • heap (堆)
  • stack (栈)

    golang程序，是一个进程
  • 一个进程对应着多个OS线程，又根据[[MPG模型]]，对应着很多个协程；
  • 对所有的goroutinue来说，heap可以看作是共享的区域
  • 但是每个goroutinue，都有只属于自己的stack

一般程序的内存管理
stack会随着线程执行code area的stack frame，自动pop、push、remove；
stack里的变量、调用完毕，就随着出栈，自动销毁；
但是heap不会，heap内的对象，通常是用stack内或者其它heap对象内的指针变量，对heap内的对象进行操作；
这个也叫做对象引用；
垃圾回收，garbage collect，回收的就是heap的空间

如何标记

在 GC 领域里，判断对象存活的主流思路是两个，「引用计数」和「可达性分析」。

  1. 引用计数
    顾名思义，引用计数的思路就是给每个对象进行计数，每被其它对象引用一次，计数就 +1，引用失效后，计数就 -1。当计数器的数值为 0，就意味着它没有被使用，可以回收。

  2. 可达性分析
    可达性分析的思路就是通过引用链路判断对象是否可被触达，如果能触达说明该对象当前正在被使用，不可回收;反之，没有触达到的对象则认为是无使用的，可以回收。

这个引用链路的结构类似于有向有环图，但是根节点不止一个，是一个集合，称之为 GCRoots。

目前主流的 GC 机制大多用的是「可达性分析」这条路线，我们Golang 的GC机制也是如此。

滑动窗口

  1. 滑动窗口：不断的调节子序列的起始位置和终止位置，从而得出我们要想的结果

滑动窗口也可以理解为双指针法的一种，本质上还是使用两个类指针来表示起始位置

  1. 滑动窗口常用来解决字符串的匹配问题。
  • 匹配问题套路：不满足题意，右移窗口，更新数据；满足题意，左移窗口，更新数据。
  1. 实现滑动窗口，主要确定如下四点：
    （1）初始化窗口:
    初始化左右边界 left = right = 0，把索引闭区间 [left, right] 称为一个「窗口」。
    （2）寻找可行解：
    我们先不断地增加 right 指针扩大窗口 [left, right]，直到窗口中的满足可行解。
    （3）优化可行解：
    此时，我们停止增加 right，转而不断增加 left 指针缩小窗口 [left, right]，直到窗口中的可行解不再符合要求。同时，每次增加 left，我们都要更新一轮结果。
    （4）滑动窗口，直至一次遍历结束：
    重复第 2 和第 3 步，直到 right 到达到的尽头。

    基本思想

    二分查找的基本思想是将目标值与数组中间的元素进行比较，如果目标值小于中间元素，则在数组的左半部分继续查找，否则在右半部分查找，不断缩小搜索范围，直到找到目标值或确定目标值不存在为止。

写法

写二分法，区间的定义一般为两种，左闭右闭即[left, right]，或者左闭右开即[left, right)

[left, right]

定义target在[left, right]区间

  • for (left <= right) 要使用 <=

因为right是包含在区间内的，是可以取到right值的，left == right是有意义的。所以用<=

  • if (target < arr[mid]) right = mid - 1

因为当前这个arr[mid]一定不是target，那么接下来要查找的左区间结束下标位置就是 middle - 1

在二分查找中，每次比较都将搜索空间减半，这导致了对数级别的时间复杂度。即使数据量增加，二分查找的时间增长速度也很慢，因此它是一种高效的查找算法。

计算公式

用T(n)表示比较n个元素的二分查找的时间复杂度，则有以下递归关系：T(n)=T(n/2)+1

  • T(n/2) 表示在一次比较之后，搜索空间被减半，问题规模变为原来的一半。

对于二分查找，a = 1，b = 2，f(n) = 1。
根据主定理T(n)=aT(n/b​)+f(n)，时间复杂度的表达式是：
T(n)=T(n/2)+1
T(n)=Θ(logn)

时间复杂度：O(log n)。

主定理（Master Theorem）：

T(n)=aT(n/b​)+f(n)

  • a是子问题的个数
  • b是问题规模缩小的倍数
  • f(n)是除了子问题递归调用之外的其他操作的代价。

1. 对于标准的二分查找算法，每次都将问题规模减半，因此只有一个子问题，即 a=1。
2. 每次都将搜索空间缩小一半，所以b=2
3. 每一步的操作是一个比较，所以f(n)=1

前言

因为最近在写个人博客的项目，在做项目时，涉及用户注册、登录、退出等操作，要对登陆用户的合法性进行判断，这就需要用到身份验证了。

认证、授权、鉴权是什么？

认证

认证(Identification) 是指根据声明者所特有的识别信息，确认声明者的身份
白话文的意思就是：你需要用身份证证明你自己是你自己。

比如我们常见的认证技术：

  • 身份证
  • 用户名和密码
  • 用户手机：手机短信、手机二维码扫描、手势密码
  • 用户的电子邮箱

授权

授权(Authorization)： 在信息安全领域是指资源所有者委派执行者，赋予执行者指定范围的资源操作权限，以便对资源的相关操作。

在现实生活领域例如： 银行卡（由银行派发）、门禁卡（由物业管理处派发）、钥匙（由房东派发），这些都是现实生活中授权的实现方式。

在互联网领域例如： web 服务器的 session 机制、web 浏览器的 cookie 机制、颁发授权令牌（token）等都是一个授权的机制。

鉴权

鉴权(Authentication) 在信息安全领域是指对于一个声明者所声明的身份权利，对其所声明的真实性进行鉴别确认的过程。

在现实生活领域： 门禁卡需要通过门禁卡识别器，银行卡需要通过银行卡识别器；

在互联网领域： 校验 session/cookie/token 的合法性和有效性

先授权，后鉴权

也就是说，用户必须保持有效，然后才能根据其授权级别授予对资源的访问权限。一旦通过身份验证，就会为它们分配不同的角色，从而向它们授予对系统的特殊权限。

那cookie安全吗？

我们知道 HTTP 是明文传输协议，因此如果使用 HTTP 传输 Cookie 信息，则存在以下几个问题：

  1. 容易被窃听：HTTP 请求及其内容都是以明文形式传输的，因此第三方可以轻松地截取和查看请求内容，并获取其中包含的 Cookie 信息。
  2. 容易被篡改：由于 HTTP 没有提供任何防篡改措施，因此第三方可以修改请求内容或者插入恶意代码来窃取 Cookie 信息。
    那我们的用户密码不是不安全吗，这时候我们可以使用我们前面提到的HTTPS来安全地传输敏感信息和 Cookie 数据。

Token

一般校验用户的机密，我们只让一个组件来完成，多个组件访问就可能出现企业出内鬼的问题，也就是说看守507房间的这个门卫，我不应该随随便便把我的密码告诉他。为了避免密码被反复的传来传去，我们会使用一个比密码更加安全的东西。就是token

什么是token？Token是服务端生成的一串字符串，以作客户端进行请求的一个令牌，当第一次登录后，服务器生成一个Token便将此Token返回给客户端，以后客户端只需带上这个Token前来请求数据即可，无需再次带上用户名和密码。

为什么说token比密码更加安全？

  • 密码是永久有效的
  • 令牌有时间限制，往往只在20min-2hours有效

数字签名


分为两个部分，第一个部分是完全明文的，比如I’m Zqxg Go：507 Id：0001，一般开发者喜欢用Json/base64编码,只有这一段明文还是不行的，因为黑客也可以生成出这一段明文，所以我们还要加入一段数字签名

什么是数字签名？
_数字签名_（又称公钥数字签名）是只有信息的发送者才能产生的别人无法伪造的一段数字串，这段数字串同时也是对信息的发送者发送信息真实性的一个有效证明。

简单来说就是大门门卫用自己的私钥加密上面的明文得到签名后的字符串。黑客无法访问到他的私钥，从而黑客得不到数字签名。

而当我拿着我虎符上楼的时候，每个小门门卫都知道大门门卫的公钥，他要确定的一点就是这张虎符一定是楼底大门门卫颁发的。怎么确定呢？他用大门门卫颁发的公钥去解密虎符后半部分的签名，

  • 如果解密的结果刚好等于签名明文的一部分，就证明这张虎符就是楼底大门门卫颁发的
  • 如果解密发生错误/解密结果跟明文不符合 就说明这个令牌是伪造的

需求解决

当小门门卫确定这个令牌是大门门卫颁发的之后，他才会去看里面的内容。数据中包含着我是否能进入这个小门，从而确定放我进去 or 不放我进去

黑客无法伪造这样的虎符：他无法搞到大门门卫的私钥，只能用他自己的秘钥去进行数字签名，这样的数字签名小门门卫就会发现解不开 or 解开了但是跟前面明文不符合

JWT

JWT的token包含三部分数据：

  1. Header：头部，通常头部有两部分信息：
  • 声明类型type，这里是JWT（type=jwt）
  • 加密算法，自定义(rs256/base64/hs256) 我们会对头部进行base64加密（可解密），得到第一部分数据
  1. Payload：载荷，就是有效数据，一般包含下面信息：
  • 用户身份信息-userid,username（注意，这里因为采用base64加密，可解密，因此不要存放敏感信息）
  • 注册声明：如token的签发时间，过期时间，签发人等 这部分也会采用base64加密，得到第二部分数据

（1）标准中注册的声明（建议但不强制使用）
iss: jwt签发者
sub: jwt所面向的用户
aud: 接收jwt的一方
exp: jwt的过期时间，这个过期时间必须要大于签发时间
nbf: 定义在什么时间之前，该jwt都是不可用的.
iat: jwt的签发时间
jti: jwt的唯一身份标识，主要用来作为一次性token,从而回避重放攻击。
（2）公共的声明
公共的声明可以添加任何的信息，一般添加用户的相关信息或其他业务需要的必要信息. 但不建议添加敏感信息，因为该部分在客户端可解密.
（3）私有的声明
私有声明是提供者和消费者所共同定义的声明，一般不建议存放敏感信息，因为base64 是对称解密的，意味着该部分信息可以归类为明文信息。

  1. Signature：base64加密，签名，是整个数据的认证信息。一般根据前两步的数据，再加上服务的的密钥（secret，盐）（不要泄漏，最好周期性更换），通过加密算法生成。用于验证整个数据完整和可靠性

一个JWT实际上就是一个字符串，它由三部分组成，头部、载荷与签名。

双Token

双Token是什么？指的是Access Token(访问令牌) & Refresh Token (刷新令牌)

  • 我前面讲的 Token，都是 Access Token。
  • Refresh Token：作用是刷新AccessToken。认证服务器会提供一个刷新接口，我们传入Refresh_token，认证服务器通过后会返回一个新的AccessToken。
    特点：
  1. Refresh Token 的有效期会比较长，
  2. 当 Access Token 由于过期而失效时，使用 Refresh Token 就可以获取到新的 Access Token 如果 Refresh Token 也失效了，用户就只能重新登录了

双Token流程：

  1. 用户身份验证:当用户通过用户名和密码登录时，服务器会验证用户的身份。如果验证成功，服务器会生成访问令牌和刷新令牌，并将它们返回给客户端。
  2. 使用访问令牌:客户端收到令牌后，会将访问令牌存储在本地(一般存在SessionStorage, 而刷新令牌存在LocalStorage)。然后，客户端在每次向服务器发送请求时，都会在请求头中携带访问令牌。
  3. 访问令牌过期：当访问令牌过期后，服务器会拒绝来自客户端的请求,并返回一个特定的错误码，例如401(未授权)。
  4. 使用刷新令牌:当客户端收到401错误码时,它会使用刷新令牌向服务器请求新的访问令牌。服务器会验证刷新令牌，如果验证成功，服务器会生成新的访问令牌，并将它返回给客户端。
  5. 刷新令牌过期:如果刷新令牌也过期了，客户端就需要重新进行身份验证。

 ### JWT +Refresh Token

在 JWT 的实践中，引入 Refresh Token，将会话管理流程改进如下：

  1. 客户端使用用户名密码进行认证
  2. 服务端生成有效时间较短的 Access Token（例如 10 分钟），和有效时间较长的 Refresh Token（例如 7 天）
  3. 客户端访问需要认证的接口时，携带 Access Token
  4. 如果 Access Token 没有过期，服务端鉴权后返回给客户端需要的数据
  5. 如果携带 Access Token 访问需要认证的接口时鉴权失败（例如返回 401 错误），则客户端使用 Refresh Token 向刷新接口申请新的 Access Token
  6. 如果 Refresh Token 没有过期，服务端向客户端下发新的 Access Token
  7. 客户端使用新的 Access Token 访问需要认证的接口

上述的架构，提供了服务端禁用用户 Token 的方式，当用户需要登出或禁用用户时，只需要将服务端的 Refresh Token 禁用或删除，用户就会在 Access Token 过期后，由于无法获取到新的 Access Token 而再也无法访问需要认证的接口。这样的方式虽然会有一定的窗口期（取决于 Access Token 的失效时间），但是结合用户登出时客户端删除 Access Token 的操作，基本上可以适应常规情况下对用户认证鉴权的精度要求。
